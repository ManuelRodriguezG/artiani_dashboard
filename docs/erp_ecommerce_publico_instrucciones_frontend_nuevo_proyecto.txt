======================================================================
1. OBJETIVO DEL NUEVO PROYECTO FRONTEND
======================================================================

Crear un proyecto frontend separado del ERP para mostrar un catalogo publico vivo de productos para mascotas.

Este proyecto NO debe ser un checkout completo en Fase 1.

Objetivo Fase 1:
- Mostrar productos publicados desde el ERP.
- Mostrar imagen, nombre, marca, categoria, presentacion, precio y disponibilidad publica simple.
- Permitir filtros por mascota y necesidad.
- Permitir busqueda rapida.
- Permitir carrito local tipo cotizacion.
- Validar carrito contra el ERP con endpoint dry-run.
- Generar mensaje para WhatsApp.
- No cobrar online.
- No descontar inventario.
- No crear pedido confirmado.
- No usar tablas legacy ecom_* como fuente.

Idea de negocio:
- No construir una tienda generica.
- Construir un ecosistema alrededor de mascotas.
- Ayudar al cliente a encontrar rapido productos para su mascota.
- Preparar el camino para futuras mascotas registradas, recomendaciones, recordatorios, recompensas, alianzas con veterinarias y campanas.

======================================================================
2. ARQUITECTURA GENERAL
======================================================================

El ERP es la fuente de verdad.

El frontend ecommerce sera un proyecto separado que consume API publica del ERP.

Responsabilidades del ERP:
- Catalogo real.
- SKU, producto, marca, categoria, imagenes, presentacion.
- Precio vivo.
- Disponibilidad publica simple.
- Configuracion publica del canal.
- Cotizacion dry-run.
- Futuro registro interno de cotizacion/prospecto.

Responsabilidades del frontend:
- Experiencia visual publica.
- Busqueda y filtros.
- Carrito local.
- UX de cotizacion.
- Llamadas a API del ERP.
- Armado de mensaje WhatsApp usando datos recalculados por ERP.

No hacer desde frontend:
- Leer base de datos.
- Leer tablas internas del ERP.
- Consumir tablas o endpoints legacy ecom_*.
- Calcular precio final como fuente de verdad.
- Mostrar stock exacto.
- Descontar inventario.
- Crear pedidos confirmados.
- Procesar pagos.

6. CONTRATOS FUNCIONALES DE API
======================================================================

GET /estado

Uso:
- Saber si el ERP ya esta listo para datos reales.
- Si ready=false, mostrar "catalogo en preparacion".

Campos importantes:
- depurar.ready
- depurar.schema.ddl_pendiente
- depurar.publicaciones.publicadas
- depurar.configuracion.whatsapp_configurado
- depurar.configuracion.cors_configurado

GET /contratos

Uso:
- Validar version y endpoints.

Validar:
- depurar.api.version debe ser fase1-2026-07-12.

GET /configuracion

Uso:
- Obtener moneda, numero WhatsApp, mensaje base y politicas publicas.

No hardcodear WhatsApp en frontend.

GET /filtros

Uso:
- Obtener mascotas, necesidades, marcas y categorias disponibles.

GET /catalogo

Parametros opcionales:
- q
- mascota
- necesidad
- marca
- categoria
- pagina
- limite

Uso:
- Mostrar grilla/listado de productos publicados.

Nota:
- Antes de datos reales puede regresar vacio y eso no es error fatal.

GET /producto/{slug}

Uso:
- Mostrar detalle de producto publicado.

GET /disponibilidad

Uso:
- Consultar disponibilidad publica simple.

Estados esperados:
- disponible
- pocas_piezas
- consultar_disponibilidad
- agotado

POST /cotizacion_dryrun

Uso:
- Revalidar carrito antes de WhatsApp.
- Recalcular precios y totales desde ERP.
- No escribe BD.
- No descuenta inventario.
- No crea pedido.

Body ejemplo:

{
  "items": [
    {
      "id_publicacion": 1,
      "slug": "producto-ejemplo",
      "id_sku": 123,
      "cantidad": 2
    }
  ],
  "contacto": {
    "nombre": "Cliente",
    "telefono": "",
    "mensaje": ""
  },
  "utm": {
    "source": "web"
  }
}

Regla:
- El frontend puede guardar carrito local, pero debe confiar en el resultado del dry-run para precio, subtotal y mensaje final.

7. SEGURIDAD Y CORS
======================================================================

Estado Fase 1:
- CORS esta cerrado por defecto.
- El ERP solo permitira origenes configurados en erp_ecommerce_configuracion.cors_origenes_permitidos.
- API key/HMAC esta documentado como seguridad futura, no requerido para read-only local.

Cuando el frontend tenga host local, configurar en ERP:

cors_origenes_permitidos=http://localhost:3000

Si usa otro puerto:

cors_origenes_permitidos=http://localhost:5173

Nunca usar:

Access-Control-Allow-Origin: *

Produccion:
- Usar dominio exacto del ecommerce.
- No exponer secretos en navegador.
- Si hay HMAC, implementarlo en backend del ecommerce.
- No habilitar POST real sin rate limit/captcha/politica de seguimiento.

Headers futuros documentados:

X-Ecommerce-Api-Key
X-Ecommerce-Timestamp
X-Ecommerce-Nonce
X-Ecommerce-Signature

En Fase 1 no construir dependencia dura de esos headers si el ERP aun esta en modo read-only local.
