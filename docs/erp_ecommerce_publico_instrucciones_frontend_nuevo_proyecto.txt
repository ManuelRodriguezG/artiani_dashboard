======================================================================
1. OBJETIVO DEL NUEVO PROYECTO FRONTEND
======================================================================

Crear un proyecto frontend separado del ERP para mostrar un catalogo publico vivo de productos para mascotas.

Este proyecto NO debe ser un checkout completo en Fase 1.

Objetivo Fase 1:
- Mostrar productos publicados desde el ERP.
- Mostrar imagen, nombre, marca, categoria, presentacion, precio y disponibilidad publica simple.
- Permitir filtros por mascota y necesidad.
- Permitir busqueda rapida.
- Permitir carrito local tipo cotizacion.
- Validar carrito contra el ERP con endpoint dry-run.
- Generar mensaje para WhatsApp.
- No cobrar online.
- No descontar inventario.
- No crear pedido confirmado.
- No usar tablas legacy ecom_* como fuente.

Idea de negocio:
- No construir una tienda generica.
- Construir un ecosistema alrededor de mascotas.
- Ayudar al cliente a encontrar rapido productos para su mascota.
- Preparar el camino para futuras mascotas registradas, recomendaciones, recordatorios, recompensas, alianzas con veterinarias y campanas.

======================================================================
2. ARQUITECTURA GENERAL
======================================================================

El ERP es la fuente de verdad.

El frontend ecommerce sera un proyecto separado que consume API publica del ERP.

Responsabilidades del ERP:
- Catalogo real.
- SKU, producto, marca, categoria, imagenes, presentacion.
- Precio vivo.
- Disponibilidad publica simple.
- Configuracion publica del canal.
- Cotizacion dry-run.
- Futuro registro interno de cotizacion/prospecto.

Responsabilidades del frontend:
- Experiencia visual publica.
- Busqueda y filtros.
- Carrito local.
- UX de cotizacion.
- Llamadas a API del ERP.
- Armado de mensaje WhatsApp usando datos recalculados por ERP.

No hacer desde frontend:
- Leer base de datos.
- Leer tablas internas del ERP.
- Consumir tablas o endpoints legacy ecom_*.
- Calcular precio final como fuente de verdad.
- Mostrar stock exacto.
- Descontar inventario.
- Crear pedidos confirmados.
- Procesar pagos.

GET /configuracion

Uso:

GET /seo

Uso:
- Obtener metadatos para title/description, robots, sitemap y JSON-LD.

- No inventar URLs canonicas si `url_sitio_publico` aun no esta configurado.

GET /filtros

Uso:
- Obtener mascotas, necesidades, marcas y categorias disponibles.

GET /catalogo

Parametros opcionales:
- q
- mascota
- necesidad
- marca
- categoria
- pagina
- limite

Uso:
- Mostrar grilla/listado de productos publicados.

Nota:
- Antes de datos reales puede regresar vacio y eso no es error fatal.

GET /producto/{slug}

Uso:
- Mostrar detalle de producto publicado.

GET /disponibilidad

Uso:
- Consultar disponibilidad publica simple.

Estados esperados:
- disponible
- pocas_piezas
- consultar_disponibilidad
- agotado

Uso:
- Revalidar carrito antes de WhatsApp.

- No descuenta inventario.

Regla:

7. SEGURIDAD Y CORS
======================================================================

Estado Fase 1:
- CORS esta cerrado por defecto para origenes no configurados.
- El ERP permite `http://artiani.com.local` porque esta configurado en `erp_ecommerce_configuracion.cors_origenes_permitidos`.
- El ERP solo debe permitir origenes exactos configurados; nunca wildcard.
- API key/HMAC esta documentado como seguridad futura, no requerido para read-only local.

Origen local configurado actualmente:

cors_origenes_permitidos=http://artiani.com.local

Si usa otro puerto/host, agregar ese origen exacto:

cors_origenes_permitidos=http://localhost:5173

Nunca usar:

Access-Control-Allow-Origin: *

Produccion:
- Usar dominio exacto del ecommerce.
- No exponer secretos en navegador.
- Si hay HMAC, implementarlo en backend del ecommerce.
- No habilitar POST real sin rate limit/captcha/politica de seguimiento.

Headers futuros documentados:

X-Ecommerce-Api-Key
X-Ecommerce-Timestamp
X-Ecommerce-Nonce
X-Ecommerce-Signature

En Fase 1 no construir dependencia dura de esos headers si el ERP aun esta en modo read-only local.

- detalle de producto;
- carrito local tipo cotizacion;
- validacion de carrito via `/cotizacion_dryrun`;
- envio por WhatsApp cuando el ERP tenga configuracion.

No construir checkout, pasarela, pagos, pedidos confirmados ni descuento de inventario en Fase 1.

## Reglas de arquitectura

- El ERP es la fuente de verdad.
- Este frontend no lee BD.
- Este frontend no consume tablas internas.
- Este frontend no consume legacy `ecom_*`.
- Todo dato de catalogo viene de `/ecommercePublico`.
- Precios y disponibilidad se revalidan con el ERP antes de WhatsApp.
- El carrito local no descuenta inventario.
- El frontend no crea pedidos ni ventas.

## Endpoints permitidos

- `GET /contratos`
- `GET /estado`
- `GET /configuracion`
- `GET /filtros`

No usar todavia:

- `POST /cotizacion_registrar`

## Seguridad

- CORS debe usar origenes exactos configurados en el ERP.
- No usar `Access-Control-Allow-Origin: *`.
- No guardar secretos HMAC/API secret en navegador.
- Si en el futuro se requiere firma HMAC, implementarla en backend del ecommerce.

## UX

- La primera pantalla debe ser el catalogo usable, no una landing generica.
- La UI debe sentirse de negocio de mascotas, no tienda generica.
- Priorizar busqueda rapida, filtros claros y carrito simple.
- No mostrar stock exacto.
- Disponibilidad publica: Disponible, Pocas piezas, Consultar disponibilidad, Agotado.
- Usar "cotizacion" y "total estimado sujeto a confirmacion".
- No decir "pedido confirmado", "pagado" o "compra finalizada".

## Estados obligatorios

- API no disponible.
- API disponible pero `ready=false`.
- Catalogo en preparacion.
- Catalogo vacio.
- Producto no encontrado.
- Producto agotado.
- WhatsApp no configurado.
- Cotizacion dry-run con bloqueos.
